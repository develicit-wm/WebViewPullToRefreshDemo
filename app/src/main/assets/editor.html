Refreshed!
게시글 작성

안녕하세요, 이 글은 Pull-to-Refresh 테스트용 게시글입니다.

이 데모 앱은 Android WebView에서 SwipeRefreshLayout과 함께 사용할 때 발생할 수 있는 다양한 스크롤 충돌 문제를 테스트하기 위해 만들어졌습니다.

테스트 시나리오

1. 상단에서 Pull-to-Refresh: 페이지 최상단에서 아래로 당기면 새로고침이 동작해야 합니다.

2. 스크롤 중 Pull-to-Refresh 방지: 페이지를 스크롤한 상태에서는 Pull-to-Refresh가 동작하지 않아야 합니다.

3. 에디터 내부 스크롤: contenteditable 영역 내에서 텍스트를 편집하고 스크롤할 때 Pull-to-Refresh가 간섭하지 않아야 합니다.

4. 입력 중 새로고침 방지: 사용자가 텍스트를 입력하는 중에는 실수로 새로고침이 발생하지 않아야 합니다.

알려진 이슈들

WebView와 SwipeRefreshLayout을 함께 사용할 때 가장 흔한 문제는 스크롤 좌표 충돌입니다. WebView 내부의 스크롤 위치를 SwipeRefreshLayout이 올바르게 감지하지 못하면, 페이지 중간에서도 Pull-to-Refresh가 트리거될 수 있습니다.

이를 해결하기 위해 일반적으로 WebView.getScrollY() 값을 확인하거나, JavaScript를 통해 window.scrollY 또는 document.scrollingElement.scrollTop 값을 네이티브로 전달하는 방식을 사용합니다.

1. 스크롤 가능한 리스트 overflow-y

4. 이중 중첩 스크롤 nested overflow
외부 컨테이너 안에 내부 스크롤 영역이 있는 구조. 각각 독립적으로 스크롤되어야 합니다.

외부 영역 텍스트 1: SwipeRefreshLayout과 WebView를 함께 사용할 때, 중첩된 스크롤 뷰가 있으면 터치 이벤트 처리가 매우 복잡해집니다.

외부 영역 텍스트 2: Android의 NestedScrolling 메커니즘은 부모와 자식 뷰 간의 스크롤 협력을 지원하지만, WebView 내부의 CSS overflow 스크롤은 이 시스템 밖에서 동작합니다.

내부 텍스트 1: 중첩 스크롤의 핵심은 터치 이벤트가 올바른 스크롤 컨테이너에 전달되는 것입니다.

내부 텍스트 2: CSS의 overscroll-behavior 속성을 사용하면 스크롤 체이닝을 제어할 수 있습니다.

내부 텍스트 3: 하지만 Android WebView에서는 이 속성의 지원이 불완전할 수 있습니다.

내부 텍스트 4: 따라서 JavaScript로 터치 이벤트를 직접 처리하는 것이 더 안정적일 수 있습니다.

내부 텍스트 5: touchstart, touchmove, touchend 이벤트를 감시하여 스크롤 방향과 위치를 계산합니다.

내부 텍스트 6: 이 데이터를 바탕으로 preventDefault()를 호출할지 결정합니다.

내부 텍스트 7: 추가 더미 텍스트입니다.

내부 텍스트 8: 스크롤이 충분히 가능하도록 텍스트를 추가합니다.

외부 영역 텍스트 3: 내부 컨테이너 이후의 콘텐츠입니다. 내부 스크롤이 끝까지 도달한 후 외부로 전파되는지 확인하세요.

외부 영역 텍스트 4: 이 부분까지 스크롤이 도달했다면 외부 컨테이너의 스크롤이 정상 동작하는 것입니다.

Pull-to-Refresh 구현 가이드

Android에서 WebView와 SwipeRefreshLayout을 함께 사용할 때는 몇 가지 핵심 사항을 고려해야 합니다.

1. 스크롤 위치 감지
SwipeRefreshLayout은 자식 뷰의 canScrollVertically(-1) 메서드를 호출하여 위로 스크롤이 가능한지 판단합니다. WebView의 경우 이 메서드가 항상 정확한 값을 반환하지 않을 수 있으므로, 커스텀 로직이 필요할 수 있습니다.

2. JavaScript Interface 활용
WebView 내부의 정확한 스크롤 위치를 알기 위해 JavaScript Interface를 사용할 수 있습니다. window.scrollY 값을 네이티브 코드로 전달하면 더 정확한 제어가 가능합니다.

3. Touch Event 처리
복잡한 레이아웃에서는 터치 이벤트의 전파 순서를 정확히 이해하고 제어해야 합니다. onInterceptTouchEvent와 onTouchEvent의 동작 방식을 숙지하는 것이 중요합니다.

4. 성능 최적화
Pull-to-Refresh 동작 중에는 가능한 한 메인 스레드의 부하를 줄여야 합니다. 새로고침 시 네트워크 요청은 비동기로 처리하고, UI 업데이트는 최소화해야 합니다.

5. 접근성 고려
Pull-to-Refresh 제스처가 접근성 서비스를 사용하는 사용자에게도 올바르게 동작하는지 확인해야 합니다. TalkBack 등의 스크린 리더와 호환되는지 테스트가 필요합니다.

6. Edge Cases
빠른 스크롤 후 즉시 Pull-to-Refresh를 시도하는 경우, 두 손가락으로 동시에 터치하는 경우, 화면 회전 중 Pull-to-Refresh가 진행 중인 경우 등 다양한 엣지 케이스를 고려해야 합니다.

이 글은 충분한 스크롤 길이를 확보하기 위한 더미 콘텐츠를 포함하고 있습니다. 실제 앱에서는 이 영역에 API로부터 받은 동적 콘텐츠가 표시됩니다.

추가적으로, WebView 내부의 contenteditable 영역에서 편집 중인 경우에도 Pull-to-Refresh가 의도치 않게 동작하지 않도록 주의해야 합니다. 이는 특히 게시글 작성, 댓글 입력 등의 기능에서 중요합니다.
